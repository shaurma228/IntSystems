## Задание 1 — `1MO.py`

### Кратко

Первая практика строит базовую систему прогнозирования оттока клиентов телеком-компании. Скрипт загружает `telecom_churn.csv`, изучает структуру данных, разделяет признаки и целевую переменную `Churn`, кодирует категориальные признаки, масштабирует числовые и обучает три модели: логистическую регрессию, дерево решений и k-ближайших соседей. Затем сравнивает их по `Accuracy`, выбирает лучшую модель и строит для нее матрицу ошибок.

### Подробное объяснение кода

В начале подключаются библиотеки:

- `pandas` нужен для чтения CSV и работы с таблицей.
- `matplotlib.pyplot` и `seaborn` используются для графиков.
- `ColumnTransformer` позволяет по-разному обработать числовые и категориальные признаки.
- `StandardScaler` стандартизирует числовые признаки.
- `OneHotEncoder` превращает категориальные столбцы в набор бинарных признаков.
- `LogisticRegression`, `DecisionTreeClassifier`, `KNeighborsClassifier` — три модели для сравнения.
- `accuracy_score` считает долю правильных ответов.
- `confusion_matrix` строит матрицу ошибок.
- `train_test_split` делит данные на обучение и тест.

Далее строка `df = pd.read_csv('telecom_churn.csv')` загружает датасет. `df.head()` печатает первые 5 строк, чтобы быстро посмотреть, как выглядят данные.

Блок исследования данных выводит:

- `df.shape` — количество строк и столбцов;
- `df.info()` — типы столбцов и наличие пропусков;
- `df.describe()` — статистику числовых признаков;
- `df.isnull().sum()` — количество пропусков в каждом столбце.

Целевая переменная — `Churn`. Команда `df['Churn'].value_counts()` показывает, сколько клиентов осталось и сколько ушло. `df['Churn'].mean() * 100` работает потому, что `Churn` обычно закодирован как `0/1`: среднее значение равно доле единиц, то есть доле ушедших клиентов.

Далее данные делятся на признаки и целевую переменную:

- `X = df.drop('Churn', axis=1)` — все столбцы, кроме `Churn`;
- `y = df['Churn']` — только целевой столбец.

`train_test_split(X, y, test_size=0.25, random_state=42, stratify=y)` делит данные на 75% train и 25% test. `random_state=42` делает разбиение воспроизводимым. `stratify=y` сохраняет примерно одинаковую долю ушедших и оставшихся клиентов в train и test.

Затем явно задаются категориальные признаки:

- `State`;
- `Area code`;
- `International plan`;
- `Voice mail plan`.

Все остальные признаки считаются числовыми через list comprehension:

```python
numerical_features = [col for col in X.columns if col not in categorical_features]
```

`ColumnTransformer` создает общий предобработчик:

- для числовых признаков применяется `StandardScaler`;
- для категориальных — `OneHotEncoder(drop='first', handle_unknown='ignore')`.

`drop='first'` удаляет одну категорию из каждого категориального признака, чтобы уменьшить избыточность. `handle_unknown='ignore'` защищает код от новых категорий в тесте.

Важно: `preprocessor.fit_transform(X_train)` обучает преобразования только на train и сразу преобразует train. `preprocessor.transform(X_test)` применяет уже обученные преобразования к test. Это правильно, потому что тестовые данные не должны влиять на обучение.

Модели лежат в словаре `models`. Цикл:

```python
for name, model in models.items():
    model.fit(X_train_processed, y_train)
    y_pred = model.predict(X_test_processed)
    acc = accuracy_score(y_test, y_pred)
```

по очереди обучает каждую модель, делает прогноз на тесте и считает `Accuracy`.

Результаты сохраняются в `results`, затем превращаются в `DataFrame`. По нему строится barplot, где визуально видно, какая модель точнее.

Лучшая модель выбирается строкой:

```python
best_model_name = max(results, key=results.get)
```

Она берет ключ словаря с максимальным значением accuracy.

Для лучшей модели строится матрица ошибок:

```python
cm = confusion_matrix(y_test, y_pred_best)
tn, fp, fn, tp = cm.ravel()
```

В бинарной классификации:

- `TN` — модель верно сказала, что клиент остался;
- `FP` — модель ошибочно сказала, что клиент ушел;
- `FN` — модель ошибочно сказала, что клиент остался, хотя он ушел;
- `TP` — модель верно сказала, что клиент ушел.

Для бизнеса особенно важны `FN`: это ушедшие клиенты, которых модель не заметила.

## Задание 2 — `2MO.py`

### Кратко

Вторая практика берет лучшую модель из первой работы, подробно диагностирует ее качество и пытается улучшить результат. Скрипт снова готовит данные, определяет модель-победителя, строит `classification_report` и матрицу ошибок, анализирует дисбаланс классов, создает `Pipeline`, добавляет борьбу с дисбалансом через `class_weight='balanced'` или `SMOTE`, подбирает гиперпараметры через `GridSearchCV`, считает финальные метрики, строит ROC-кривую и график важности признаков.

### Подробное объяснение кода

Код снова загружает `telecom_churn.csv`, отделяет `X` и `y`, делает `train_test_split` с `stratify=y`. Это нужно, чтобы сравнение с первой практикой было честным: данные делятся так же.

Дальше повторяется подготовка:

- категориальные признаки перечисляются явно;
- числовые находятся автоматически;
- `ColumnTransformer` масштабирует числовые признаки и кодирует категориальные.

Сначала код заново обучает три модели из первой практики и выбирает лучшую по accuracy. Это имитирует ситуацию “загрузили модель-победитель из прошлой практики”, хотя фактически модель пересоздается и переобучается.

После выбора лучшей модели выполняется диагностика:

- `classification_report` показывает `precision`, `recall`, `f1-score` и `support` для классов `0` и `1`;
- `ConfusionMatrixDisplay` строит матрицу ошибок.

`precision` показывает, насколько можно доверять положительным прогнозам модели. `recall` показывает, какую долю настоящих ушедших клиентов модель нашла. Для оттока часто важнее `recall` класса `1`, потому что пропустить уходящего клиента дороже, чем ошибочно предложить скидку лояльному.

Дальше анализируется дисбаланс:

```python
print(y_train.value_counts())
print(y_train.mean() * 100)
```

Если ушедших клиентов мало, модель может хорошо угадывать большинство, но плохо находить меньшинство.

Затем создается новый `ColumnTransformer`. Это правильный подход: предобработка должна быть частью единого конвейера.

Код определяет класс лучшей модели:

```python
model_class = type(best_model)
```

Если модель поддерживает `class_weight`, создается обычный `sklearn.pipeline.Pipeline`:

```python
Pipeline([
    ('preprocessor', preprocessor),
    ('classifier', model_class(class_weight='balanced', random_state=42))
])
```

`class_weight='balanced'` увеличивает вес редкого класса, чтобы модель сильнее обращала внимание на ушедших клиентов.

Если модель не поддерживает `class_weight`, используется `SMOTE`. Он синтетически создает новые объекты редкого класса в тренировочных данных. Для SMOTE нужен `imblearn.pipeline.Pipeline`, потому что SMOTE должен применяться только внутри обучения, а не к тесту.

Для подбора параметров создается `param_grid`. Он зависит от типа модели:

- для `LogisticRegression` перебираются `C`, `penalty`, `solver`;
- для `DecisionTreeClassifier` — `max_depth`, `min_samples_split`, `min_samples_leaf`;
- для `KNeighborsClassifier` — `n_neighbors`, `weights`, `p`.

`GridSearchCV` перебирает все комбинации параметров, делает 5-кратную кросс-валидацию и выбирает лучшую по `scoring='f1'`. F1 выбран потому, что балансирует precision и recall.

После `grid_search.fit(X_train, y_train)` доступны:

- `best_params_` — лучшие параметры;
- `best_score_` — средний результат на кросс-валидации;
- `best_estimator_` — готовый лучший pipeline.

Финальная модель оценивается на test:

- `Accuracy`;
- `Precision`;
- `Recall`;
- `F1-score`;
- `ROC-AUC`.

Для ROC-кривой нужны вероятности класса `1`:

```python
y_proba = best_final_model.predict_proba(X_test)[:, 1]
```

`roc_curve` возвращает точки кривой, а `roc_auc_score` считает площадь под ней. Чем ближе AUC к 1, тем лучше модель отделяет ушедших клиентов от лояльных.

Блок важности признаков работает по-разному:

- если у модели есть `coef_`, значит это линейная модель, и важность берется по модулю коэффициентов;
- если есть `feature_importances_`, значит это дерево, и используются встроенные важности признаков.

`get_feature_names_out()` получает имена признаков после `OneHotEncoder`, чтобы график был читаемым.

В конце создается таблица сравнения старой и новой модели по ключевым метрикам и дополнительно строится матрица ошибок финальной модели.

## Задание 3 — `3MO.py`

### Кратко

Третья практика работает с классическим датасетом `Wine` из `sklearn`. Нужно определить тип винограда/вина по химическим показателям. Код загружает данные, изучает число классов и признаков, смотрит распределение классов, делит данные на train/test со stratify, обучает `RandomForestClassifier`, `LogisticRegression` и `RandomForestClassifier(class_weight='balanced')`, а затем сравнивает accuracy, `classification_report`, матрицы ошибок и macro-F1.

### Подробное объяснение кода

В начале импортируются:

- `datasets` для загрузки `load_wine`;
- `RandomForestClassifier` и `LogisticRegression` — модели классификации;
- `classification_report`, `confusion_matrix`, `ConfusionMatrixDisplay`, `accuracy_score` — метрики и визуализация;
- `Pipeline` и `StandardScaler` — для масштабирования перед логистической регрессией.

`RANDOM_STATE = 42` фиксирует случайность, `TEST_SIZE = 0.25` задает размер тестовой выборки.

Функция `print_section` просто печатает красивый заголовок раздела. Она нужна, чтобы вывод программы был читаемым.

Функция `analyze_class_distribution` принимает `y` и названия классов. Внутри:

```python
counts = np.bincount(y, minlength=len(target_names))
```

считает, сколько объектов каждого класса есть в выборке. Затем выводится таблица: номер класса, название, количество и доля. `np.argmin(counts)` находит самый редкий класс, а `counts.max() / counts.min()` показывает силу дисбаланса.

Функция `describe_worst_class` анализирует, какой класс модель предсказывает хуже всего. Она берет recall каждого класса из `classification_report(..., output_dict=True)`. Худший класс — тот, у которого recall минимальный. Затем по матрице ошибок считается:

- сколько объектов этого класса было в test;
- сколько найдено верно;
- сколько пропущено.

Функция `evaluate_model` объединяет обучение и оценку одной модели:

1. Печатает название модели.
2. Обучает модель через `model.fit(X_train, y_train)`.
3. Делает прогноз `model.predict(X_test)`.
4. Считает accuracy.
5. Печатает `classification_report`.
6. Строит матрицу ошибок численно и графически.
7. Вызывает `describe_worst_class`.
8. Возвращает словарь с метриками.

В `main()` загружается датасет:

```python
wine = datasets.load_wine()
X = wine.data
y = wine.target
```

`X` — матрица химических признаков вина, `y` — номер класса вина. `target_names` содержит названия классов, `feature_names` — названия химических показателей.

После EDA выполняется:

```python
train_test_split(..., stratify=y)
```

`stratify=y` важен: он сохраняет пропорции классов в train и test.

Словарь `models` содержит три варианта:

- обычный `RandomForest`;
- `LogisticRegression` внутри `Pipeline(StandardScaler + LogisticRegression)`;
- `RandomForest balanced` с `class_weight='balanced'`.

Логистическая регрессия получает масштабированные признаки, потому что она чувствительна к разным масштабам признаков. RandomForest масштабирования не требует.

После обучения всех моделей создается таблица сравнения:

- `accuracy`;
- `macro_f1`;
- худший класс;
- recall худшего класса.

`macro_f1` особенно важен при многоклассовой классификации: он считает F1 отдельно по каждому классу и усредняет их одинаково, без учета размера класса. Поэтому он хорошо показывает баланс качества между классами.

Затем сравнивается обычный RandomForest и RandomForest с `class_weight='balanced'`. Смысл `class_weight` — дать больший вес редким классам, чтобы модель меньше их пропускала.

Итоговый вывод выбирает модель, которая лучше балансирует классы и лучше подходит, если важно не пропустить редкий класс.

## Задание 4 — логика `4MO.py`

### Кратко

Четвертая практика должна улучшать macro-F1 лучшей модели из третьей практики. Основные инструменты: `SMOTE`, кривые обучения и `GridSearchCV`. Идея такая: сначала берется базовая модель из практики 3, затем к train применяется SMOTE, после чего сравнивается macro-F1. Потом строится learning curve для диагностики переобучения/недообучения и запускается подбор гиперпараметров RandomForest.

### Как должен быть устроен код

Сначала загружается тот же `Wine`:

```python
wine = datasets.load_wine()
X = wine.data
y = wine.target
```

Разбиение должно быть тем же, что в третьей практике:

```python
train_test_split(X, y, test_size=..., random_state=42, stratify=y)
```

Это важно: если поменять test, сравнение с практикой 3 будет нечестным.

Базовая модель должна повторять лучшую модель из практики 3. Если была выбрана `RandomForest balanced`, значит стартовая точка:

```python
RandomForestClassifier(random_state=42, class_weight='balanced')
```

Далее применяется SMOTE только к тренировочным данным:

```python
smote = SMOTE(random_state=42)
X_train_smote, y_train_smote = smote.fit_resample(X_train, y_train)
```

Нельзя применять SMOTE к test, потому что test должен имитировать реальные новые данные.

После SMOTE обучается RandomForest и считается `classification_report` и macro-F1 на test. Сравнение идет так:

- macro-F1 базовой модели;
- macro-F1 модели после SMOTE;
- изменение recall худшего/редкого класса.

Для кривых обучения используется `learning_curve`. Она обучает модель на разных размерах train и считает качество на train и validation. Если train сильно выше validation — есть переобучение. Если обе кривые низкие — модель недообучена. Если validation растет с увеличением данных — может помочь сбор новых данных.

Для комплексной настройки создается `Pipeline`. Если внутри есть SMOTE, правильнее использовать `imblearn.pipeline.Pipeline`:

```python
Pipeline([
    ('scaler', StandardScaler()),
    ('smote', SMOTE(random_state=42)),
    ('classifier', RandomForestClassifier(random_state=42))
])
```

Затем задается сетка:

```python
{
    'classifier__max_depth': [5, 10, None],
    'classifier__n_estimators': [50, 100]
}
```

`GridSearchCV` запускается с:

- `StratifiedKFold(n_splits=3)`;
- `scoring='f1_macro'`;
- pipeline в качестве estimator.

После обучения берется `best_estimator_`, оценивается на test и строится итоговая таблица:

- базовая модель из практики 3;
- модель после SMOTE/class_weight;
- модель после GridSearchCV.

## Задание 5 — `5MO.py`

### Кратко

Пятая практика переходит от классификации к регрессии. Нужно предсказывать стоимость домов по датасету `California Housing`. Код изучает целевую переменную, строит гистограмму цен, выбирает 3 понятных признака, визуализирует их связь с ценой, обучает `LinearRegression`, считает `MAE`, `MSE`, `R2`, объясняет коэффициенты линейной модели и сравнивает результат с `DecisionTreeRegressor(max_depth=3)`.

### Подробное объяснение кода

Импорты:

- `fetch_california_housing` загружает датасет;
- `LinearRegression` — линейная модель;
- `DecisionTreeRegressor` — дерево решений для регрессии;
- `mean_absolute_error`, `mean_squared_error`, `r2_score` — метрики регрессии;
- `matplotlib` строит графики.

`SELECTED_FEATURES = ["MedInc", "AveRooms", "HouseAge"]` задает три признака:

- `MedInc` — медианный доход района;
- `AveRooms` — среднее число комнат;
- `HouseAge` — возраст домов.

`FEATURE_DESCRIPTIONS` хранит человекочитаемые описания признаков. Они используются в подписях графиков и при объяснении коэффициентов.

Функция `load_dataset` загружает данные:

- `X` — все признаки;
- `y` — целевая переменная `MedHouseVal`;
- `feature_names` — названия признаков.

`MedHouseVal` измеряется в сотнях тысяч долларов. Значение `2.5` означает примерно 250 000 долларов.

Функция `get_selected_columns` находит индексы выбранных признаков и возвращает только эти столбцы:

```python
selected_indices = [feature_names.index(feature) for feature in SELECTED_FEATURES]
return X[:, selected_indices], selected_indices
```

Функция `plot_target_histogram` строит гистограмму целевой переменной. По ней видно распределение цен: где объектов много, где мало, есть ли ограничение сверху.

Функция `plot_feature_relationships` строит scatter plot для каждого выбранного признака. По оси X — значение признака, по оси Y — цена. Это помогает визуально понять, есть ли связь между признаком и ценой.

Функция `evaluate_regression_model` универсальна для любой регрессионной модели:

1. Обучает модель.
2. Делает прогноз.
3. Считает `MAE`, `MSE`, `R2`.
4. Возвращает модель, прогнозы и метрики.

`MAE` — средняя абсолютная ошибка. Ее проще объяснять бизнесу: модель ошибается в среднем на столько-то сотен тысяч долларов.

`MSE` — средняя квадратичная ошибка. Она сильнее штрафует большие ошибки.

`R2` — доля объясненной вариации. Чем ближе к 1, тем лучше. Значение около 0 означает, что модель почти не лучше среднего прогноза.

Функция `money_text` переводит коэффициент из сотен тысяч долларов в обычные доллары для красивого вывода.

Функция `explain_linear_coefficients` печатает:

- `model.coef_` — коэффициенты при признаках;
- `model.intercept_` — свободный член.

Смысл коэффициента: если признак увеличивается на 1, то прогноз цены меняется на значение коэффициента при прочих равных. Например, если коэффициент при `AveRooms` равен `0.10`, то увеличение среднего числа комнат на 1 связано с ростом прогнозируемой цены примерно на 10 000 долларов.

В `main()` сначала загружаются данные и выводится статистика целевой переменной: минимум, среднее, медиана, максимум.

Затем строятся графики:

- гистограмма `MedHouseVal`;
- scatter plot для `MedInc`, `AveRooms`, `HouseAge`.

После этого данные делятся:

```python
train_test_split(X_selected, y, test_size=0.25, random_state=42)
```

Обучается `LinearRegression`, затем `DecisionTreeRegressor(max_depth=3)`.

Функция `compare_models` сравнивает модели по таблице метрик и выбирает лучшую по `R2`.

Последний график — реальные цены против предсказанных. Если модель идеальна, точки лежат на линии `y=x`. Чем сильнее разброс вокруг линии, тем хуже прогноз.

## Задание 6 — `6MO.py`

### Кратко

Шестая практика показывает переобучение и регуляризацию. Код берет данные из прошлой практики, обучает базовую линейную регрессию, затем добавляет 30 случайных шумовых признаков. На зашумленных данных обычная линейная регрессия может начать подгоняться под шум. Для исправления используется `Ridge`-регрессия, которая штрафует слишком большие коэффициенты и помогает уменьшить переобучение.

### Подробное объяснение кода

Импорты:

- `fetch_california_housing` загружает данные;
- `LinearRegression` — базовая модель;
- `Ridge` — линейная регрессия с L2-регуляризацией;
- `r2_score` — метрика качества;
- `train_test_split` — деление на train/test;
- `StandardScaler` — масштабирование признаков;
- `numpy` — работа с массивами и генерация шума.

Константы:

- `TEST_SIZE = 0.3` — 30% данных уходит в test;
- `SELECTED_FEATURES` — те же признаки, что в практике 5;
- `N_NOISE_FEATURES = 30` — количество случайных шумовых признаков.

В одной из версий файла есть `DEMO_TRAIN_SIZE = 45`. Это учебный прием: на полном California Housing объектов много, поэтому 30 шумовых признаков не всегда сильно портят качество. Маленький train делает переобучение заметнее.

Функция `load_previous_practice_data` загружает California Housing и выбирает только признаки из 5 практики:

```python
selected_indices = [feature_names.index(feature) for feature in SELECTED_FEATURES]
return X[:, selected_indices], y, SELECTED_FEATURES
```

Функция `scale_train_test` создает `StandardScaler`, обучает его на train и применяет к train/test:

```python
X_train_scaled = scaler.fit_transform(X_train)
X_test_scaled = scaler.transform(X_test)
```

Это правильный порядок: scaler обучается только на train, чтобы test не влиял на подготовку данных.

Функция `add_noise_features` создает случайные признаки:

```python
train_noise = rng.normal(size=(X_train.shape[0], n_noise_features))
test_noise = rng.normal(size=(X_test.shape[0], n_noise_features))
```

Эти признаки не связаны с ценой домов. Затем они добавляются к исходным признакам через `np.hstack`.

Функция `train_linear_and_get_r2` обучает `LinearRegression` и возвращает:

- саму модель;
- `R2_train`;
- `R2_test`.

Сравнение `R2_train` и `R2_test` важно для диагностики переобучения. Если `R2_train` высокий, а `R2_test` сильно ниже, модель выучила особенности train, которые не работают на новых данных.

Функция `train_ridge_and_get_test_r2` делает то же самое, но для `Ridge(alpha=alpha)`.

`alpha` — сила регуляризации:

- маленький `alpha` почти не ограничивает модель;
- большой `alpha` сильнее сжимает коэффициенты;
- слишком большой `alpha` может привести к недообучению.

В `main()` сначала загружаются данные, делятся на train/test:

```python

```

Затем чистые данные масштабируются, обучается базовая `LinearRegression`, и сохраняется `baseline_test_r2`.

После этого добавляются 30 шумовых признаков, новые данные снова масштабируются, и обучается новая `LinearRegression`. Это “сломанная” модель.

Затем перебираются значения:

```python
alphas = [0.1, 1, 10, 100, 1000]
```

Для каждого `alpha` обучается Ridge и выводится `R2_train`, `R2_test`. Лучший `alpha` выбирается по максимальному `R2_test`:

```python

```

Функция `print_results_table` печатает итоговую таблицу:

- базовая модель;
- сломанная модель с шумом;
- исправленная Ridge-модель.

Также она отвечает на вопросы:

- насколько упало качество после добавления шума;
- при каком `alpha` Ridge оказался лучшим;
- насколько Ridge приблизился к базовому уровню;
- уменьшился ли разрыв между train и test.

Главная идея задания: обычная линейная регрессия может поставить ненулевые коэффициенты случайным шумовым признакам. Ridge добавляет штраф за большие коэффициенты и делает модель устойчивее.

## Что полезно запомнить по всем заданиям

- `train_test_split` отделяет данные для честной проверки.
- `random_state=42` нужен для воспроизводимости.
- `stratify=y` важен в классификации при дисбалансе классов.
- `StandardScaler` обучается только на train.
- `OneHotEncoder` нужен для категориальных признаков.
- `Pipeline` помогает не забыть одинаковую предобработку на train и test.
- `GridSearchCV` подбирает гиперпараметры через кросс-валидацию.
- `classification_report` удобен для классификации.
- `MAE`, `MSE`, `R2` используются для регрессии.
- `Ridge` помогает бороться с переобучением линейной модели.
